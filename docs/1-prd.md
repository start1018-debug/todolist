# TodoList 앱 PRD

## 1. 개요

### 목적

TodoList 앱은 개인 사용자가 자신의 할 일을 간단하게 등록, 완료 처리, 삭제, 조회할 수 있는 웹 애플리케이션이다.

### 배경

혼자 작업하거나 공부하는 사용자는 복잡한 협업 도구보다 빠르게 할 일을 기록하고 완료 여부를 관리할 수 있는 단순한 도구가 필요하다.

본 앱은 회원가입과 로그인 기반으로 개인별 할 일 목록을 안전하게 관리하는 것을 목표로 한다. Supabase는 PostgreSQL 데이터베이스 용도로만 사용하며, 인증은 백엔드에서 자체 JWT 방식으로 구현한다.

### 기술 스택

| 영역 | 기술 |
| --- | --- |
| 프론트엔드 | React + Vite + TypeScript |
| 프론트엔드 배포 | Vercel |
| 백엔드 | Node.js + Express + JavaScript |
| 백엔드 DB 접근 | `pg` 라이브러리 |
| 백엔드 배포 | Vercel |
| 데이터베이스 | Supabase PostgreSQL |
| 인증 | 백엔드 자체 JWT 인증 |

## 2. 타깃 사용자

- 혼자 할 일을 관리하고 싶은 개인 사용자
- 개인 프로젝트나 학습 일정을 정리하려는 1인 개발자
- 복잡한 협업 기능 없이 간단한 체크리스트가 필요한 사용자

## 3. 사용자 스토리

### 3.1 회원가입

나는 개인 사용자로서 내 할 일 데이터를 안전하게 저장하기 위해 회원가입을 원한다.

Acceptance Criteria:

- 사용자는 이메일과 비밀번호로 회원가입할 수 있다.
- 이미 가입된 이메일로는 회원가입할 수 없다.
- 회원가입 성공 후 로그인 페이지로 이동하거나 로그인 상태가 된다.

### 3.2 로그인

나는 개인 사용자로서 내 할 일 목록에 접근하기 위해 로그인을 원한다.

Acceptance Criteria:

- 사용자는 이메일과 비밀번호로 로그인할 수 있다.
- 로그인 성공 시 백엔드에서 자체 JWT 토큰이 발급된다.
- 로그인하지 않은 사용자는 할 일 목록 화면과 Todo API에 접근할 수 없다.

### 3.3 할 일 추가

나는 개인 사용자로서 해야 할 일을 기록하기 위해 할 일 추가 기능을 원한다.

Acceptance Criteria:

- 사용자는 텍스트 입력 후 할 일을 추가할 수 있다.
- 빈 텍스트 또는 공백만 있는 텍스트로는 할 일을 추가할 수 없다.
- 추가된 할 일은 현재 로그인한 사용자에게만 표시된다.

### 3.4 완료 체크

나는 개인 사용자로서 진행 상태를 확인하기 위해 할 일 완료 체크 기능을 원한다.

Acceptance Criteria:

- 사용자는 각 할 일의 완료 여부를 체크하거나 해제할 수 있다.
- 완료된 할 일은 체크 상태와 텍스트 스타일 등으로 시각적으로 구분된다.
- 완료 상태 변경 후 새로고침해도 상태가 유지된다.

### 3.5 할 일 삭제

나는 개인 사용자로서 필요 없는 항목을 정리하기 위해 할 일 삭제 기능을 원한다.

Acceptance Criteria:

- 사용자는 자신의 할 일을 삭제할 수 있다.
- 삭제된 할 일은 목록에서 즉시 사라진다.
- 사용자는 다른 사용자의 할 일을 삭제할 수 없다.

### 3.6 전체 목록 조회

나는 개인 사용자로서 내 할 일을 한눈에 보기 위해 전체 목록 조회 기능을 원한다.

Acceptance Criteria:

- 로그인한 사용자는 자신의 전체 할 일 목록을 조회할 수 있다.
- 목록에는 할 일 내용, 완료 여부, 생성일이 표시된다.
- 다른 사용자의 할 일은 조회되지 않는다.

## 5. 비기능 요구사항

### 5.1 반응형 웹 UI

- 모바일 360px 이상, 태블릿, 데스크톱 화면을 모두 지원한다.
- TailwindCSS의 `sm`, `md`, `lg` 브레이크포인트를 기준으로 레이아웃을 전환한다.
- 모바일에서는 단일 컬럼 중심의 간결한 UI를 제공한다.
- 태블릿과 데스크톱에서는 입력 영역과 목록 영역이 안정적으로 표시되도록 최대 너비, 여백, 정렬 방식을 조정한다.
- 버튼, 입력창, Todo 항목의 텍스트가 작은 화면에서도 겹치거나 잘리지 않아야 한다.

### 5.2 성능

- 초기 화면 로딩 시 불필요한 리소스와 API 호출을 최소화한다.
- 할 일 추가, 완료 변경, 삭제 요청 후 UI는 사용자가 변화를 즉시 인지할 수 있게 갱신되어야 한다.
- 목록 조회 API는 로그인한 사용자의 데이터만 조회하며 필요한 필드만 반환한다.
- 일반적인 개인 사용량 기준으로 Todo 목록 조회는 1초 이내 응답을 목표로 한다.

### 5.3 보안

- 인증은 Supabase Auth를 사용하지 않고 백엔드 자체 JWT 방식으로 구현한다.
- 비밀번호는 서버에서 안전한 해시 알고리즘으로 해시 처리 후 저장한다.
- JWT는 서버에서 검증하며, 인증이 필요한 API는 토큰 없이 접근할 수 없다.
- 사용자는 자신의 할 일 데이터에만 접근할 수 있어야 한다.
- 클라이언트에서 전달된 사용자 ID를 신뢰하지 않고 JWT의 사용자 정보를 기준으로 처리한다.
- SQL 쿼리는 `pg`의 파라미터 바인딩을 사용해 SQL Injection 위험을 줄인다.

### 5.4 접근성

- 버튼, 입력창, 체크박스는 키보드로 조작 가능해야 한다.
- 입력 필드에는 명확한 라벨 또는 접근 가능한 이름을 제공한다.
- 완료 상태는 색상만이 아니라 체크 상태나 텍스트 스타일로도 구분한다.
- 주요 에러 메시지는 사용자가 인지할 수 있는 위치에 표시한다.
- 기본 텍스트와 배경은 충분한 명도 대비를 가져야 한다.

### 5.5 배포 및 운영

- 프론트엔드와 백엔드는 Vercel 배포를 기준으로 구성한다.
- 백엔드는 Supabase PostgreSQL에 연결하기 위한 환경 변수를 사용한다.
- JWT secret, DB connection string 등 민감 정보는 클라이언트에 노출하지 않는다.
- Supabase는 PostgreSQL 데이터베이스 기능만 사용하며 Supabase Auth, Storage, Realtime 등은 사용하지 않는다.

| Todo | 사용자가 등록한 개별 할 일 항목 |
