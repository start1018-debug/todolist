# TodoList 앱 와이어프레임

## 1. 화면 도출 요약

PRD의 사용자 스토리와 `docs/3-api-spec.yaml` 기준으로 필요한 화면은 다음 4개이다.

| 화면 | 주요 목적 | 관련 사용자 스토리 | 주요 API |
| --- | --- | --- | --- |
| 인증 진입 화면 | 비로그인 사용자를 로그인 또는 회원가입으로 안내 | 3.1 회원가입, 3.2 로그인 | 없음 |
| 로그인 화면 | 이메일/비밀번호로 JWT 발급 | 3.2 로그인 | `POST /auth/login` |
| 회원가입 화면 | 이메일/비밀번호로 계정 생성 및 JWT 발급 | 3.1 회원가입 | `POST /auth/register` |
| Todo 목록 화면 | Todo 전체 조회, 추가, 완료 체크, 삭제 | 3.3 할 일 추가, 3.4 완료 체크, 3.5 할 일 삭제, 3.6 전체 목록 조회 | `GET /todos`, `POST /todos`, `PATCH /todos/{todoId}/completed`, `DELETE /todos/{todoId}` |

## 2. 공통 아키텍처 전제

- 프론트엔드: React + Vite + TypeScript
- 스타일: TailwindCSS 반응형 브레이크포인트 `sm`, `md`, `lg`
- 인증: 백엔드 자체 JWT
- API Base URL: `http://localhost:3000/api` 또는 배포 환경의 API URL
- 인증 토큰 저장: 새로고침 후 로그인 상태 유지를 위해 클라이언트 저장소에 `accessToken` 저장
- 보호 라우트: `accessToken`이 없거나 Todo API에서 `401`이 오면 로그인 화면으로 이동
- API 요청 헤더: Todo API 호출 시 `Authorization: Bearer {accessToken}` 적용
- 서버 신뢰 기준: 사용자 ID는 클라이언트 입력값이 아니라 JWT에서 확인한 사용자 ID 기준으로 처리

## 3. 인증 진입 화면

- 3.2 로그인: 사용자는 이메일과 비밀번호로 로그인할 수 있다.
- 3.2 로그인: 로그인하지 않은 사용자는 할 일 목록 화면과 Todo API에 접근할 수 없다.

### 사용자 인터랙션 흐름

1. 사용자가 앱에 처음 진입한다.
2. 클라이언트가 저장된 `accessToken` 존재 여부를 확인한다.

4. 토큰이 없으면 인증 진입 화면을 보여준다.
5. 사용자가 `로그인`을 누르면 로그인 화면으로 이동한다.
6. 사용자가 `회원가입`을 누르면 회원가입 화면으로 이동한다.

- 3.2 로그인: 로그인 성공 시 백엔드에서 자체 JWT 토큰이 발급된다.
- 3.2 로그인: 로그인하지 않은 사용자는 할 일 목록 화면과 Todo API에 접근할 수 없다.

### 호출하는 API 엔드포인트

- `POST /auth/login`
  - Request: `{ email, password }`
  - Success: `accessToken`, `tokenType`, `user`
  - Error: `400 ValidationError`, `401 INVALID_CREDENTIALS`, `500 InternalServerError`

### 컴포넌트 트리


### 사용자 인터랙션 흐름

1. 사용자가 이메일과 비밀번호를 입력한다.
2. `로그인` 버튼을 누르거나 Enter 키로 제출한다.
3. 클라이언트가 이메일 형식과 필수값을 1차 검증한다.
4. 검증 실패 시 입력 필드 아래에 오류 메시지를 표시하고 API를 호출하지 않는다.
5. 검증 성공 시 `POST /auth/login`을 호출한다.
6. 요청 중에는 버튼을 비활성화하고 로딩 상태를 표시한다.
7. 성공하면 `accessToken`을 저장하고 Todo 목록 화면으로 이동한다.
8. `401` 응답이면 로그인 실패 메시지를 표시한다.
9. `500` 응답이면 잠시 후 다시 시도하라는 오류 메시지를 표시한다.

### 반응형 레이아웃 변화

- 모바일: 화면 폭 대부분을 사용하는 단일 폼, 입력창과 버튼은 전체 너비다.
- 태블릿: 폼 최대 너비를 `420px` 전후로 제한하고 중앙 정렬한다.
- 데스크톱: 인증 폼을 중앙에 배치하고 여백을 넓혀 반복 사용 시 집중도를 높인다.

9. 사용자가 `로그인` 링크를 누르면 로그인 화면으로 이동한다.

- 3.3 할 일 추가: 사용자는 텍스트 입력 후 할 일을 추가할 수 있다.
- 3.3 할 일 추가: 빈 텍스트 또는 공백만 있는 텍스트로는 할 일을 추가할 수 없다.
- 3.3 할 일 추가: 추가된 할 일은 현재 로그인한 사용자에게만 표시된다.
- 3.4 완료 체크: 사용자는 각 할 일의 완료 여부를 체크하거나 해제할 수 있다.
- 3.4 완료 체크: 완료 상태 변경 후 새로고침해도 상태가 유지된다.
- 3.5 할 일 삭제: 사용자는 자신의 할 일을 삭제할 수 있다.
- 3.5 할 일 삭제: 사용자는 다른 사용자의 할 일을 삭제할 수 없다.
- 3.6 전체 목록 조회: 로그인한 사용자는 자신의 전체 할 일 목록을 조회할 수 있다.
- 3.6 전체 목록 조회: 목록에는 할 일 내용, 완료 여부, 생성일이 표시된다.

### 호출하는 API 엔드포인트

- `GET /todos`
  - 화면 진입 시 호출한다.
  - 완료 여부 필터를 구현하는 경우 `GET /todos?completed=true|false`를 사용할 수 있다.
  - 성공 시 `Todo[]`를 렌더링한다.
- `POST /todos`
  - Todo 추가 시 호출한다.
  - Request: `{ title }`
  - 성공 시 응답 Todo를 목록 상단에 추가하거나 `GET /todos`를 재호출한다.
- `PATCH /todos/{todoId}/completed`
  - 체크박스 변경 시 호출한다.
  - Request: `{ completed }`
  - 성공 시 해당 Todo의 완료 상태와 `updatedAt`을 갱신한다.
- `DELETE /todos/{todoId}`
  - 삭제 버튼 클릭 시 호출한다.
  - `204` 성공 시 해당 Todo를 목록에서 제거한다.

### 사용자 인터랙션 흐름

1. 사용자가 Todo 목록 화면에 진입한다.
2. `accessToken`이 없으면 로그인 화면으로 이동한다.
3. `accessToken`이 있으면 `GET /todos`를 호출한다.
4. 요청 중에는 목록 영역에 로딩 상태를 표시한다.
5. 성공 응답의 `data`가 빈 배열이면 빈 상태를 표시한다.
6. 성공 응답의 `data`가 있으면 Todo 항목을 생성일 기준으로 표시한다.
7. 사용자가 새 Todo 제목을 입력한다.
8. 제목이 빈 문자열 또는 공백뿐이면 오류 메시지를 표시하고 `POST /todos`를 호출하지 않는다.
9. 제목이 유효하면 `POST /todos`를 호출한다.
10. 생성 성공 시 입력창을 비우고 새 Todo를 목록에 반영한다.
11. 사용자가 체크박스를 클릭하면 `PATCH /todos/{todoId}/completed`를 호출한다.
12. 완료 변경 요청 중에는 해당 체크박스를 비활성화하거나 낙관적 UI 후 실패 시 원복한다.
13. 완료된 Todo는 체크 상태와 취소선 또는 낮은 대비 텍스트로 구분한다.
14. 사용자가 삭제 버튼을 누르면 삭제 확인이 필요한 경우 확인 후 `DELETE /todos/{todoId}`를 호출한다.
15. 삭제 성공 시 해당 Todo를 즉시 목록에서 제거한다.
16. Todo API에서 `401`이 오면 저장된 토큰을 제거하고 로그인 화면으로 이동한다.
17. Todo API에서 `404`가 오면 이미 삭제되었거나 접근할 수 없는 항목으로 보고 목록을 갱신한다.
18. Todo API에서 `500`이 오면 목록 또는 항목 단위 오류 메시지를 표시하고 재시도 버튼을 제공한다.

### 반응형 레이아웃 변화

- 모바일: 헤더, 입력 영역, 목록 영역을 모두 세로 단일 컬럼으로 배치한다. Todo 항목은 제목과 생성일을 줄바꿈해 표시하고 삭제 버튼은 우측 또는 하단에 둔다.
- 태블릿: 입력 영역은 상단 전체 폭, 목록은 최대 너비가 있는 중앙 컬럼으로 유지한다.
- 데스크톱: 입력 영역과 목록 영역을 2컬럼으로 배치할 수 있다. 좌측은 새 Todo 입력, 우측은 Todo 목록으로 나누어 반복 작업 시 목록 스캔이 쉽도록 한다.
- 모든 화면: 긴 Todo 제목은 줄바꿈하고, 체크박스와 삭제 버튼은 고정 크기로 유지해 텍스트 길이에 따라 레이아웃이 흔들리지 않게 한다.

## 7. 상태별 공통 UI

### 로딩 상태

```text
┌────────────────────────────┐
│ Todo 목록을 불러오는 중...  │
└────────────────────────────┘
```

- 적용 화면: Todo 목록 화면
- API: `GET /todos`
- 목적: PRD의 로딩 상태 표시 요구사항 반영

### 빈 상태

```text
┌────────────────────────────┐
│ 아직 등록된 할 일이 없음    │
│ 위 입력창에서 새 할 일을 추가│
└────────────────────────────┘
```

- 적용 화면: Todo 목록 화면
- 조건: `GET /todos` 성공 응답의 `data`가 빈 배열
- 목적: PRD의 빈 상태 표시 요구사항 반영

### 오류 상태

```text
┌────────────────────────────┐
│ 목록을 불러오지 못했습니다  │
│ [ 다시 시도 ]               │
└────────────────────────────┘
```

- 적용 화면: 로그인, 회원가입, Todo 목록 화면
- API: 전체 API
- 목적: 입력값 검증과 주요 에러 메시지 표시 요구사항 반영

## 8. 라우팅 구조

| Route | 화면 | 접근 조건 |
| --- | --- | --- |
| `/` | 인증 진입 화면 또는 Todo 목록 화면 리다이렉트 | 토큰 있으면 `/todos`, 없으면 인증 진입 |
| `/login` | 로그인 화면 | 비로그인 사용자 |
| `/register` | 회원가입 화면 | 비로그인 사용자 |
| `/todos` | Todo 목록 화면 | JWT 필요 |

## 9. 접근성 고려사항

- 이메일, 비밀번호, Todo 입력창에는 명확한 `label` 또는 접근 가능한 이름을 제공한다.
- 로그인, 회원가입, 추가, 삭제 버튼은 키보드로 포커스 및 실행 가능해야 한다.
- Todo 완료 상태는 색상만으로 구분하지 않고 체크박스 상태와 텍스트 스타일을 함께 사용한다.
- API 오류 메시지는 관련 입력 필드 또는 화면 상단의 인지 가능한 위치에 표시한다.
- 삭제 버튼은 각 Todo 제목과 연결되는 접근 가능한 이름을 가진다.
